# Live data audit — Arena V4 (2026-04-26)

**Cutoff:** all queries filter `>= '2026-04-26 14:20:00+00'` (PR #280 V4 merge + 4-min buffer).
**Window observed:** 2026-04-26 14:20Z → 17:11Z (~2h51m of arena V4 wall-clock).
**Source:** read-only Supabase MCP. Every number below names the table + filter so a future query is reproducible.

## Sample-size disclaimer (read first)

This is **not** a statistical audit. The window is ~3 hours, the data is dominated by Wray's solo testing (one player accounts for 86% of points earned), and zone activity is uneven (one zone has 26 rounds, another has 4). The audit's job is **shape, not significance** — what kinds of events are firing, which zones/cards/effects are getting reps, where the data ends and instrumentation stops.

The headline use of this doc: **decide what to instrument before the balance simulator runs**, and **flag drift patterns where current numbers look like they were tuned for an older design** (Wray's framing).

## 4. Round cadence

`zone_control_history` post-V4:

| zone | max round | snapshots | first → last | inferred avg |
|---|---|---|---|---|
| Twilight Grove | 26 | 48 | 14:27 → 17:11 (164m) | ~6.3 min/round |
| Hanwu Boglands | 17 | 21 | 14:20 → 17:11 (171m) | ~10.0 min/round |
| Ancient Ruins | 8 | 28 | 14:27 → 17:11 (164m) | ~20.5 min/round |
| Tidal Depths | 6 | 10 | 16:07 → 17:00 (53m) | ~8.8 min/round |
| Sky Citadel | 4 | 9 | 16:16 → 17:11 (55m) | ~13.8 min/round |

Caveat: `snapped_at` is when the snapshot row was written, not when the round itself ended. If snapshots fire on a fixed cron rather than on round-end, these durations are upper bounds. Twilight Grove (POISON-stack pre-applied → fast KOs → fast round resolution) gives the cleanest signal: ~6 minutes including the 35s intermission. Hanwu's 10min/round matches the stalemate pattern — fights drag because heals match damage.

Reproducible: `SELECT name, MAX(round_number), COUNT(*), MIN(snapped_at), MAX(snapped_at) FROM zone_control_history zch JOIN zones z ON z.id=zch.zone_id WHERE snapped_at >= '2026-04-26 14:20:00+00' GROUP BY name;`

## 5. Deployment lifecycle

`zone_deployments` post-V4:

- **10 total deployments**, 4 still active, 6 inactive, **0 with `ko_at` set.**

Lifetime across all data: 528 deployments total, 66 ever had `ko_at` set (12.5% lifetime KO-via-deployment-end rate).

Read this carefully: the `kos` column on `player_zone_metrics` says 13 KOs happened across V4 (from §2's table — 8 + 2 + 2 + 1 + 0). Yet zero `zone_deployments.ko_at` got written in the same window. **The two tables track different things:**

- `player_zone_metrics.kos` = "KOs *dealt by this player on this zone today*" (incremented when your fighter scores a KO).
- `zone_deployments.ko_at` = "this deployment row was retired permanently" — only set when a deployment is removed from the zone, not when a fighter is downed mid-round.

V4 round-reset respawns at full HP for the next round, so `ko_at` rarely fires in active play. The 12.5% lifetime rate is mostly from old V2/V3 data where deployments were one-shot.

**Implication:** if a future audit asks "how many deployments survive their first round," `ko_at` will not answer it. We need a new event log.

Reproducible: `SELECT COUNT(*) FILTER (WHERE ko_at IS NOT NULL), COUNT(*) FROM zone_deployments WHERE deployed_at >= '2026-04-26 14:20:00+00';`

## 6. Drift surfacing — "balance was tuned for an older design"

Three patterns in the live data look like leftovers from an earlier design.

### 6a. Zone bonuses don't match the current effect set

Current zone bonuses on `zones.house_bonus_label`:

| zone | bonus | post-V4 outcome |
|---|---|---|
| Hanwu Boglands | "All card effects +30% stronger" | stalemate — buffs damage AND heals equally |
| Sky Citadel / Tidal Depths / Sky Castle | "HEAL and BLESS +50% stronger" | only 1 KO across all three (Tidal Depths) — bonuses never tested under attack |
| Twilight Grove | "Enemies start with 1 POISON stack" | 8 KOs, 0 heals — the lethal outlier |
| Scorched Plains / Umbral Wall | "+25% max HP" | no V4 deployments yet |
| Chaos Rift | "+3% HP regen per minute" | no V4 deployments yet |
| Ancient Ruins | "All card effects +30% stronger" | 2 KOs, 9k dmg, 3.7k heals — small sample, similar heal/damage ratio to Hanwu |

Two of nine zones never see deployments; three more saw light use. The "+30% all effects" bonus actively cancels itself out (the heal/damage parity in Hanwu is the same number on both sides of the equation). The HEAL/BLESS+50% bonus has no path to become decisive in zones where nobody equips HEAL or BLESS (only 2 HEAL equips in the entire window).

This reads as a bonus design that assumed an older, more-static economy ("add +30% to whatever you cast, win zone faster") not the live-combat round-based fight where buffs apply to both fighters in the same round.

### 6b. `effect_definitions` table describes a completely different game

`effect_definitions` (24 rows, 0 in arena code paths) describes effects in **influence-economy** terms, not real-time combat:

- BURN: *"Adds flat % extra influence to your house"* — engine: HP-tick DOT
- HEAL: *"Restores X% influence to your COMMUNITY on this zone"* — engine: 7% caster maxHP to lowest-HP ally
- POISON: *"Leading COMMUNITY loses 2% influence at midnight"* — engine: 3% target maxHP per 1.5s
- BLESS: *"All COMMUNITY members casting on this zone today get +10% points"* — engine: 4% caster maxHP heal aura
- INSPIRE: *"+5 flat bonus points on this cast"* — engine: +2 ATK / +2 SPD aura

This is the **V2/V3 zone-influence design** — the era when cards were Twitter-cast manipulations of community influence percentages, not real-time combat moves. The table is dead schema; nothing reads from it. It's not load-bearing, but it's misleading reference material if any future contributor (or AI agent) treats it as canonical. Belongs in Doc B's drift report too.

### 6c. Empty event/log tables suggest planned-but-never-built logging

Empty post-V4: `casts` (0 rows), `events` (0 rows), `zone_effects` (0 rows), `card_durability_log` (0 rows), `combat_cycles` (0 rows). The schema *expected* per-cast and per-event logging at some point — none of those are wired to V4 combat. Combined with the empty cast log, **we cannot reconstruct what actually happened in any specific round** beyond the aggregated totals on `player_zone_metrics`.

## 7. Instrumentation gaps

What this audit could not answer, and why:

| Question | Blocker | Suggested fix |
|---|---|---|
| Of the 13 V4 KOs, how many were auto-attack vs DOT vs spell? | engine doesn't write per-KO source anywhere | one row per KO event in a `combat_kos` table: `{zone_id, round_number, killer_player_id, victim_player_id, source: 'auto'/'burn'/'poison'/'spell', spell_name?, occurred_at}` |
| Per-cast frequency by spell | nothing logs casts | populate the existing `casts` table from the engine, OR a thin `combat_events` insert |
| Per-tick DOT damage totals | DOT ticks aren't logged | aggregate per-round counters into `player_zone_metrics.poison_dmg`, `burn_dmg` columns |
| Round duration (real, not snapshot-timestamp) | no `round_started_at` / `round_ended_at` field | extend `zone_control_history` with timestamp pair, or write a `combat_rounds` table |
| Win/loss outcome per round | `zone_control_history.controlling_guild` shifts but doesn't pin a round-end winner | add `round_winner_guild`, `round_end_reason` ('last_standing'/'mutual_ko'/'timeout') to `zone_control_history` (or wherever `evaluateRoundEnd` writes) |

These are not blockers for the next coding session — they're the prerequisite list for a balance simulator audit (which will be the next live-data pass once it has data to chew on). Filing these as a future PR's scope, not this audit's.

## 8. What this means for the next session's work

- **For Doc B (spell catalog):** `effect_definitions` table is a third drift source on top of `EFFECTS_REFERENCE_V5.md` and the in-game card descriptions. Surface it.
- **For Doc C (voxel primitives):** the equip distribution shows ~50% of effects go unplayed in the V4 window. Don't over-design primitives for effects that may get retired or merged before the visual layer ships. Build for the 14 effects that *did* see equips first; long-tail effects can share primitives until they prove their reps.
- **For balance work (deferred):** the `zones.house_bonus_label` set needs a redesign pass *before* the simulator runs — the current bonuses are not orthogonal to round-based combat. Two bonuses (manastorm "+30% all", dawnbringer "+50% HEAL/BLESS") are the prime suspects for the heal-stalemate pathology. The POISON-stack bonus is the only one with an unambiguous live signal.
- **For instrumentation:** if the simulator initiative starts before logging is wired, it'll be running on synthetic data only — no way to compare predicted vs actual. Worth a thin `combat_kos` table as a one-PR side-quest.
