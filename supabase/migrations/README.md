# supabase/migrations/

Database migrations for the Nine Lives Network Supabase instance.

## How migrations are created

Use the Supabase MCP `apply_migration` tool from a Claude Code session. The tool:

1. Generates a timestamp-prefixed migration file in this directory.
2. Applies the SQL to the remote Supabase database immediately (not staged — applied on the spot).

Do not edit applied migration files after creation; create a new migration instead.

## Naming convention

`YYYYMMDDhhmmss_short_description.sql` — the tool handles the timestamp prefix; provide a terse snake_case description (e.g. `drop_zone_control_deprecated_columns`).

## Ordering and idempotency

Migrations run in timestamp order. Prefer idempotent SQL (`DROP ... IF EXISTS`, `CREATE ... IF NOT EXISTS`) so re-runs on fresh clones don't error.

## Relationship to `database/schema.sql`

`database/schema.sql` is a human-readable snapshot of the live schema, generated by `scripts/dump-schema.js`. It is not the source of truth — the Supabase instance is. After a migration, regenerate the snapshot with `node scripts/dump-schema.js` if you want the checked-in file to match; otherwise expect drift.

## Relationship to Drizzle

`drizzle.config.ts` and `shared/schema.ts` cover only a placeholder `users` table. Game-table schema changes go through this directory, not Drizzle. See `CLAUDE.md` → "Database: Supabase is source of truth, Drizzle is vestigial."

## Production safety for destructive migrations

`apply_migration` modifies the live database immediately. A migration that drops a column currently read by production code creates a runtime-error window until the matching code change deploys. Recommended order:

1. Land the code cleanup PR that stops reading the column.
2. Deploy that code to production.
3. Only then run `apply_migration` to drop the column.

Non-destructive additive migrations (new tables, new nullable columns) can run ahead of code safely.

### If the Supabase MCP is in read-only mode

`apply_migration` will fail with `Cannot apply migration in read-only mode.` In that case, run the migration via the Supabase dashboard SQL Editor instead, then hand-write the migration file in this directory to match what was executed. Use the current UTC timestamp as the filename prefix and open the file with a header comment noting that the migration was applied manually and why. See the 2026-04-20 `drop_zone_control_deprecated_columns` migration for the canonical example of this fallback.

## Worked example — destructive migration pattern (PR #145 / #146)

The canonical two-PR split for destructive migrations (first established by Task 4.0 slice 3; conceptually "slice 3a" and "slice 3b"):

- **PR #145** ([`feat/task-4-0-cleanup-deprecated-column-reads`](https://github.com/Mu5tbenice/nine-lives-network/pull/145)) — code cleanup only. Removes all live reads of the soon-to-be-dropped columns. Columns still exist in the schema, so the PR is independently mergeable and deployable with zero runtime risk. Merge, deploy, confirm healthy.
- **PR #146** ([`feat/task-4-0-drop-deprecated-columns`](https://github.com/Mu5tbenice/nine-lives-network/pull/146)) — runs the migration. Because the deployed code no longer reads the dropped columns, the drop is invisible at runtime.

Rationale: shipping code cleanup and the migration in a single PR creates a "small but unbounded" production window — the DB change lands immediately on `apply_migration`, but the matching code deploy depends on merge + build + deploy latency. The two-PR split collapses that window to zero.

Use this pattern whenever a migration drops or alters a column that production code currently reads.
